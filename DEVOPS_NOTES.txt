1. DEVOPS OVERVIEW
============================================================

What is DevOps?
DevOps is a set of practices that combines Software Development and IT Operations.
Its main goal is to deliver software faster, with better quality, automation, and continuous feedback.

Main goals of DevOps:
- Faster software delivery
- Better collaboration between developers and operations team
- Automation of build, test, and deployment
- Early error detection
- Reliable and repeatable deployment
- Continuous monitoring and improvement

Why DevOps is used:
Before DevOps, developers wrote code and operations teams deployed it manually.
This caused delays, errors, and communication gaps.
DevOps solves this by using automation, CI/CD pipelines, containers, monitoring, and cloud tools.

DevOps Lifecycle:
1. Plan
2. Code
3. Build
4. Test
5. Release
6. Deploy
7. Operate
8. Monitor

Real-world example:
A developer pushes code to GitHub.
GitHub Actions or Jenkins automatically builds the code, runs tests, creates a Docker image, and deploys it to a server.
If any error happens, the team gets logs or alerts.

Common DevOps tools:
- Git and GitHub: Version control
- Maven: Java build automation
- Docker: Containerization
- Docker Compose: Multi-container setup
- Jenkins: CI/CD automation
- GitHub Actions: CI/CD workflows
- Kubernetes: Container orchestration
- Terraform: Infrastructure as Code
- Ansible: Configuration management
- Prometheus and Grafana: Monitoring
- AWS/Azure/GCP: Cloud platforms

4. DOCKER
============================================================

Docker is a containerization platform.
It packages application code, dependencies, runtime, and configuration into containers.

Important terms:
Image:
A read-only template used to create containers.

Container:
A running instance of an image.

Docker Hub:
A public registry for Docker images.

Dockerfile:
A file used to build custom images.

Why Docker is used:
- Same environment everywhere
- Fast deployment
- Lightweight compared to virtual machines
- Easy scaling
- Useful for microservices
- Reduces dependency issues

Docker commands:

Check Docker version:
docker --version

List running containers:
docker ps

List all containers:
docker ps -a

List images:
docker images

Pull image:
docker pull ubuntu

Run Ubuntu container:
docker run -it ubuntu bash

Run container in background:
docker run -dit --name mycontainer ubuntu

Stop container:
docker stop mycontainer

Start container:
docker start mycontainer

Remove container:
docker rm mycontainer

Remove image:
docker rmi image-name

Execute command inside container:
docker exec -it mycontainer bash

Attach to container:
docker attach mycontainer

Copy file from host to container:
docker cp index.html container_id:/var/www/html/

Copy file from container to host:
docker cp container_id:/var/www/html/index.html .

Inspect container:
docker inspect container_id

View logs:
docker logs container_id

Real-world example:
A company runs a Node.js app inside Docker so every developer and server uses the same runtime environment.

5. DOCKERFILE
============================================================

A Dockerfile contains instructions to create a custom Docker image.

Common Dockerfile instructions:
FROM: Base image
RUN: Executes command during image build
COPY: Copies files into image
WORKDIR: Sets working directory
EXPOSE: Documents container port
CMD: Default command when container starts
ENTRYPOINT: Fixed executable for container

Example NGINX Dockerfile:

FROM nginx:latest
COPY index.html /usr/share/nginx/html/index.html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]

Build image:
docker build -t custom-nginx:v1 .

Run image:
docker run -d -p 8080:80 --name nginx-container custom-nginx:v1

RUN vs CMD:
RUN executes during image build.
CMD executes when container starts.

ENTRYPOINT vs CMD:
ENTRYPOINT defines fixed command.
CMD provides default arguments or default command.

Real-world example:
A Java application is packaged into a Docker image using Dockerfile and deployed on a cloud server.

6. DOCKER VOLUMES
============================================================

Docker volumes store persistent data.
If a container is deleted, volume data remains safe.

Create volume:
docker volume create myvolume

List volumes:
docker volume ls

Inspect volume:
docker volume inspect myvolume

Run container with volume:
docker run -dit --name vol-container -v myvolume:/data ubuntu

Write data:
docker exec -it vol-container bash
cd /data
echo "hello volume" > file.txt
exit

Remove container:
docker rm -f vol-container

Verify persistence:
docker run -dit --name new-container -v myvolume:/data ubuntu
docker exec -it new-container cat /data/file.txt

Real-world example:
MySQL data is stored in a Docker volume so database data remains safe even if the MySQL container is recreated.

7. DOCKER NETWORKING
============================================================

Docker networking allows containers to communicate.

List networks:
docker network ls

Create bridge network:
docker network create mybridge

Run containers in same network:
docker run -dit --name container1 --network mybridge ubuntu
docker run -dit --name container2 --network mybridge ubuntu

Test connectivity:
docker exec -it container1 bash
ping container2

Inspect network:
docker network inspect mybridge

Remove network:
docker network rm mybridge

Port mapping:
docker run -d -p 8080:80 nginx

Real-world example:
A backend container connects to a database container using service/container name inside the same Docker network.

8. DOCKER COMPOSE
============================================================

Docker Compose manages multi-container applications using docker-compose.yml.

Basic commands:
docker compose up
docker compose up -d
docker compose down
docker compose ps
docker compose logs
docker compose build

Example docker-compose.yml:

services:
  frontend:
    image: nginx:latest
    ports:
      - "8080:80"

  database:
    image: mysql:latest
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: devopsdb
    volumes:
      - mysql_data:/var/lib/mysql

volumes:
  mysql_data:

Real-world example:
A web application may have frontend, backend, and database containers. Docker Compose starts all services together.

9. MAVEN
============================================================

Maven is a build automation and dependency management tool for Java projects.

Maven helps to:
- Compile code
- Run tests
- Package JAR/WAR
- Manage dependencies
- Standardize project structure

Maven project structure:
src/main/java
src/main/resources
src/test/java
pom.xml
target/

Important file:
pom.xml contains project details, dependencies, plugins, and build configuration.

Maven commands:
mvn -version
mvn clean
mvn compile
mvn test
mvn package
mvn install
mvn deploy
mvn clean package
mvn clean package -DskipTests

Maven lifecycle:
validate: Checks project
compile: Compiles code
test: Runs tests
package: Creates JAR/WAR
install: Installs package locally
deploy: Uploads package to remote repository

Real-world example:
A Java Spring Boot application is built using:
mvn clean package
Then Docker image is created using the generated JAR.

10. JENKINS
============================================================

Jenkins is an open-source automation server used for CI/CD.

Jenkins can:
- Pull code from GitHub
- Build code using Maven
- Run tests
- Create artifacts
- Build Docker images
- Deploy applications
- Show pipeline status

Jenkins default port:
8080

Jenkins commands:
sudo systemctl start jenkins
sudo systemctl stop jenkins
sudo systemctl restart jenkins
sudo systemctl status jenkins
sudo systemctl enable jenkins
sudo journalctl -u jenkins

Basic Jenkins pipeline:

pipeline {
    agent any

    stages {
        stage('Clone Repository') {
            steps {
                git 'https://github.com/username/repository-name.git'
            }
        }

        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }

        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }

        stage('Deploy') {
            steps {
                echo 'Deploying application'
            }
        }
    }
}

Real-world example:
When a developer pushes code to GitHub, Jenkins webhook triggers a pipeline that builds and tests the application automatically.

11. GITHUB ACTIONS
============================================================

GitHub Actions is a CI/CD tool inside GitHub.
Workflows are written in YAML files.

Workflow path:
.github/workflows/workflow-name.yml

Example Java CI workflow:

name: Java CI with Maven

on:
  push:
    branches:
      - main

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      - name: Set up JDK 17
        uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'

      - name: Build with Maven
        run: mvn clean package

      - name: Run Tests
        run: mvn test

Real-world example:
A GitHub Actions workflow automatically checks every pull request before merging code into main branch.

12. CI/CD PIPELINE
============================================================

CI means Continuous Integration.
Developers frequently merge code into a shared repository, and automated builds/tests run.

CD means Continuous Delivery or Continuous Deployment.
Continuous Delivery means code is ready for deployment.
Continuous Deployment means code is automatically deployed.

Typical CI/CD pipeline:
1. Developer pushes code
2. Pipeline starts
3. Code checkout
4. Dependency installation
5. Build
6. Test
7. Package
8. Docker image build
9. Push image to registry
10. Deploy
11. Monitor

Real-world example:
E-commerce company uses CI/CD to deploy new features quickly without manual deployment errors.
